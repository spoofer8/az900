# 3.3 · Describe features and tools for managing and deploying Azure resources

**Domain 3 · Azure management and governance** — 30–35% of exam

Every service in Azure is manageable through several interfaces that all route to the same underlying API (**Azure Resource Manager**). The exam wants you fluent across all of them and their trade-offs.

---

## The full interaction toolkit

| Interface              | UI type         | Best for                                                                    |
| ---------------------- | --------------- | --------------------------------------------------------------------------- |
| **Azure Portal**       | Web GUI         | Exploration, learning, one-off admin tasks                                  |
| **Azure CLI**          | Command line (Bash-friendly) | Scripting, DevOps, cross-platform (Windows, macOS, Linux)      |
| **Azure PowerShell**   | Command line (PowerShell)    | Windows-centric admin scripts, cmdlet-based automation         |
| **Azure Cloud Shell**  | Browser-hosted terminal      | On-the-fly CLI/PowerShell without local install                |
| **Azure Mobile app**   | iOS / Android app            | Health checks, alerts, quick fixes on the go                   |
| **REST API**           | HTTP endpoint                | Custom tooling and non-Microsoft SDKs                          |
| **SDKs**               | Language libraries           | Programmatic control in .NET, Python, Java, JS, Go, etc.       |
| **ARM templates**      | JSON declarative             | Reproducible, versioned deployments                            |
| **Bicep**              | DSL (transpiles to ARM)      | Cleaner, easier IaC than ARM templates                         |

Every one of these ultimately calls ARM — the unified control plane.

### Every interface funnels into ARM

```
     Portal   CLI    PowerShell   Cloud Shell   Mobile   REST    SDKs    ARM/Bicep
       │      │         │            │           │       │        │         │
       ▼      ▼         ▼            ▼           ▼       ▼        ▼         ▼
    ┌────────────────────────────────────────────────────────────────────────┐
    │                Azure Resource Manager  (ARM)                           │
    │                                                                        │
    │   authenticate (Entra ID)  ·  authorise (RBAC)  ·  enforce Policy      │
    │   apply Locks  ·  log Activity  ·  orchestrate provisioning            │
    └────────────────────────────────────┬───────────────────────────────────┘
                                         │
                                         ▼
             Any Azure resource — VM, Storage account, VNet, SQL DB, Key Vault, …

    Whichever tool you use, you get the same authorisation, policy, and audit trail.
```

## Azure Portal

The web-based GUI at **`portal.azure.com`**.

- Everything Azure exposes is manageable here (though some preview features may lag)
- Customisable dashboard
- Blade-based navigation
- Cloud Shell built in

**Strengths:** discoverability, visual context, one-off tasks.
**Weaknesses:** slow for repetitive work; no version control.

---

## Azure CLI

- Cross-platform command-line tool
- Command style: `az <group> <subcommand>`, e.g. `az vm create`, `az storage account list`
- Common in bash scripts and CI/CD pipelines
- Output formats: JSON (default), table, TSV, YAML

```bash
# Examples
az login
az account show
az group create --name rg-demo --location westeurope
az vm create --resource-group rg-demo --name vm1 --image Ubuntu2204 --generate-ssh-keys
```

---

## Azure PowerShell (Az module)

- Cmdlet-style syntax: `Verb-Noun`, e.g. `New-AzResourceGroup`, `Get-AzVM`
- Native on Windows, runs on macOS/Linux with PowerShell Core
- Familiar to Windows admins used to PowerShell

```powershell
Connect-AzAccount
New-AzResourceGroup -Name rg-demo -Location westeurope
New-AzVM -ResourceGroupName rg-demo -Name vm1 -Image Ubuntu2204
```

### CLI vs PowerShell

| Aspect                   | Azure CLI                        | Azure PowerShell                     |
| ------------------------ | -------------------------------- | ------------------------------------ |
| Syntax style             | `az verb noun` (bash-like)       | `Verb-Noun` (PowerShell cmdlets)     |
| Cross-platform           | Yes                              | Yes (PowerShell Core)                |
| Object handling          | JSON output                      | Rich .NET objects, pipeline-friendly |
| Preferred by             | Devs, Linux admins               | Windows admins, PS-heavy shops       |

Both are functionally equivalent — same capabilities, different taste.

---

## Azure Cloud Shell

- Browser-based shell at **`shell.azure.com`** (also in portal, mobile app, VS Code)
- Two flavours:
  - **Bash** (Azure CLI, common Linux tools)
  - **PowerShell** (Az module + PS tools)
- Pre-authenticated as your Azure identity
- Includes: `git`, `docker` CLI, `kubectl`, `helm`, `terraform`, editors (vim, nano, code)
- Requires an Azure Storage account to persist your `$HOME` (created on first use)
- Free to use — you pay only for the tiny storage account

---

## Azure Mobile app

- iOS and Android
- Monitor status, respond to alerts, run Cloud Shell commands from your phone
- Good for on-call rotations

## Infrastructure as Code

### ARM templates
- **JSON** declarative documents
- Describe *what* you want; ARM figures out *how* to make it happen
- Idempotent — apply the same template repeatedly, end state is identical
- Verbose and heavy on brackets

### Bicep
- Microsoft's cleaner DSL for Azure IaC
- **Transpiles to ARM** — same engine, same behaviour, better syntax
- Concise, typed, modular
- Preferred over raw ARM for new projects

```bicep
resource sa 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: 'mystorage${uniqueString(resourceGroup().id)}'
  location: resourceGroup().location
  sku: { name: 'Standard_LRS' }
  kind: 'StorageV2'
}
```

Equivalent ARM template would be ~30 lines of JSON.

### ARM template vs Bicep — pick

| Prefer ARM if…                                          | Prefer Bicep if…                                    |
| ------------------------------------------------------- | --------------------------------------------------- |
| You already have a large ARM library                    | You're starting fresh                               |
| You need portability with non-Microsoft tooling         | You want the cleanest Azure-native IaC syntax       |
| You must work with something that expects JSON directly | You want type-safety and modules                    |

### Why IaC at all
- Reproducibility across environments (dev / test / prod)
- Version control (Git)
- Peer review of infra changes
- Rollback via source control
- Fast disaster recovery

## Azure Arc

Extends Azure's control plane **outside** Azure — to on-prem, other clouds, and edge.

### The Arc reach

```
                             ┌────────────────────────────┐
                             │   Azure Portal / ARM       │
                             │  Policy · RBAC · Defender  │
                             │  Monitor · Update Mgmt     │
                             └──────────────┬─────────────┘
                                            │  Arc agent installed
                                            │  on each target
                    ┌───────────────────────┼─────────────────────────┐
                    │                       │                         │
                    ▼                       ▼                         ▼
          ┌─────────────────┐   ┌───────────────────────┐   ┌─────────────────┐
          │   On-prem       │   │   Other public cloud  │   │   Edge / IoT    │
          │                 │   │                       │   │                 │
          │  Windows /Linux │   │   AWS EC2 servers     │   │  Kubernetes at  │
          │  servers        │   │   GCP Compute Engine  │   │  the edge       │
          │  Kubernetes     │   │   Any K8s cluster     │   │                 │
          │  SQL MI, Postgres│  │                       │   │                 │
          └─────────────────┘   └───────────────────────┘   └─────────────────┘

     Non-Azure resources appear IN THE AZURE PORTAL and are subject to Azure
     Policy / RBAC / Defender — via Arc. One control plane, many locations.
```

### What Arc can manage

| Resource type              | With Arc                                                       |
| -------------------------- | -------------------------------------------------------------- |
| **Servers**                | Windows / Linux VMs anywhere become manageable via Azure Portal, get Azure Policy, Defender, Update Management |
| **Kubernetes clusters**    | Any CNCF-conformant K8s (on-prem, EKS, GKE) manageable via Azure |
| **Azure data services**    | Run SQL Managed Instance and PostgreSQL anywhere with Arc      |
| **Azure App Service** (preview) | Run PaaS-like apps on Arc-enabled Kubernetes anywhere    |

### When Arc matters

- Multi-cloud governance (Azure Policy across AWS, GCP, on-prem)
- Extending Defender for Cloud protection off-Azure
- Unified single-pane-of-glass administration
- Applying Azure RBAC / Policy to non-Azure resources

> **Exam-hot** — If a scenario mentions **on-prem servers or another cloud that must appear in the Azure portal and follow Azure Policy**, the answer is Azure Arc.

## Service lifecycle in Azure

Every Azure service moves through a staged release cycle. Knowing the stage tells you what you can (and can't) rely on.

```
   Private Preview  ──▶  Public Preview  ──▶  General Availability (GA)  ──▶  Deprecation
   ──────────────       ──────────────       ───────────────────────────      ────────────
   Invite-only          Anyone with a         Production-ready:               Customer notice
   limited feedback     subscription can      full SLA, all in-scope          served; feature
   from Microsoft       opt in                regions, standard support       eventually retires
```

### What's true of Public Preview

- **No SLA** — Microsoft is not obligated to a specific uptime
- May have **limited region coverage**
- May lack **standard support** channels
- **Feature set and design may still change** before GA
- Browse and enable via `preview.portal.azure.com` or the standard portal preview toggle
- Free to try, but production use is at your own risk

### GA and beyond

- **GA** — the service is production-ready, SLA-backed, and rolls out to the announced regions
- **Deprecation / retirement** — Microsoft gives advance notice (usually 12 months for major services) and provides a migration path

> **Exam-hot** — "Which statement is true of a service in Public Preview?" — no SLA, may lack support, feature set may change before GA.

## Gotchas — deployment & management tools

- **Cloud Shell needs a storage account** — set up on first use.
- **CLI and PowerShell are equivalent in capability** — pick by preference, not by feature.
- **ARM ≠ ARM templates.** ARM is the *control plane*. ARM templates are the *IaC files*. Both live under the same name.
- **Bicep still deploys through ARM.** It's syntactic sugar, not a separate engine.
- **Azure Arc is not a management tool by itself** — it's the on-ramp that makes non-Azure resources manageable *through* the standard Azure tools.

---

## Anchor facts

- Every Azure interface ultimately hits **ARM**.
- **Portal, CLI, PowerShell, Cloud Shell, Mobile app, REST, SDKs, ARM templates, Bicep** — nine ways to interact with Azure.
- **ARM templates = JSON. Bicep = DSL. Both = idempotent IaC on the ARM engine.**
- **Cloud Shell = browser terminal, pre-authenticated, needs storage.**
- **Azure Arc = extend Azure control plane to on-prem / other clouds / edge.**
