# 3.2 · Describe features and tools in Azure for governance and compliance

**Domain 3 · Azure management and governance** — 30–35% of exam

The trickiest domain because of near-synonyms: **Policy vs RBAC**, **locks vs Policy denials**, **Purview vs Service Trust Portal**. A crisp mental model is the fastest way to answer.

---

## Microsoft Purview

Microsoft's unified **data governance** platform (across Azure, on-prem, other clouds, and SaaS).

### What it does

| Capability            | Purpose                                                                              |
| --------------------- | ------------------------------------------------------------------------------------ |
| **Data Map**          | Automatically scan and catalogue data sources across your estate                     |
| **Data Catalog**      | Business glossary + searchable index of data assets                                 |
| **Classification**    | Auto-label PII / financial / health data using built-in and custom rules            |
| **Data lineage**      | Trace how data flows from source through transformations                            |
| **Insights & risk**   | Compliance and risk reporting; sensitive-info discovery                             |

**Use it when:** you need to know *where* your sensitive data is, *who owns it*, and *how it moves.*

> **Gotcha** — Purview governs **data**. Azure Policy governs **resource configuration**. Different tools, different scope.

### Purview vs Policy — different scopes

```
   Azure Policy                              Microsoft Purview
   ─────────────                             ─────────────────
   Governs resource CONFIGURATION            Governs the DATA inside resources

   "Storage must enforce HTTPS."             "Which blobs contain PII?"
   "VM SKUs must be D-series only."          "Trace this data's lineage."
   "Every RG must have Environment tag."     "Classify the customer table."

              ▼ enforced by ARM                        ▼ scans + catalogues
   ┌───────────────────────────┐             ┌──────────────────────────┐
   │  Resource metadata,       │             │  Content of blobs,       │
   │  SKU, region, tags,       │             │  DB rows, file shares,   │
   │  extensions               │             │  SaaS data               │
   └───────────────────────────┘             └──────────────────────────┘
      Governance layer                          Data-governance layer

   Use both — they answer completely different questions.
```

## Azure Policy

Enforces organisational standards by evaluating and enforcing rules on Azure resources.

### Core concepts

| Concept                | Meaning                                                                                       |
| ---------------------- | --------------------------------------------------------------------------------------------- |
| **Policy definition**  | A single rule (JSON) — e.g. "Allowed locations = West Europe, UK South"                       |
| **Initiative**         | A group of related policy definitions — e.g. an ISO 27001 baseline                            |
| **Assignment**         | Applying a definition or initiative to a scope (MG / Sub / RG)                                |
| **Effect**             | What Policy does when a resource violates the rule                                            |

### Common effects

| Effect                | Behaviour                                                                            |
| --------------------- | ------------------------------------------------------------------------------------ |
| **Audit**             | Log a violation but allow the action                                                 |
| **Deny**              | Block the action from completing                                                     |
| **Append**            | Add fields to a resource at creation (e.g. required tags)                            |
| **Modify**            | Modify resource properties (e.g. add tags to existing resources)                     |
| **DeployIfNotExists** | Deploy a related resource when the target is created (e.g. add diagnostic settings)  |
| **AuditIfNotExists**  | Audit if a related resource does not exist                                           |
| **Disabled**          | Policy definition is inactive                                                        |

### Common built-in policies

- Allowed locations
- Allowed VM SKUs
- Require tag and value
- Storage accounts should require secure transfer (HTTPS)
- Diagnostic settings must be enabled

### From rule to enforcement — the pipeline

```
    Policy definition           Initiative                Assignment
    ─────────────────           ──────────                ──────────
    One rule (JSON):            Group of related          Attach a definition
    "Allowed locations =        definitions — e.g.        or initiative to a
     WestEU, UKSouth"           "ISO 27001 baseline"      scope: MG / Sub / RG

    ┌───────────────┐  ────▶   ┌────────────────┐  ────▶  ┌──────────────────┐
    │  Definition   │           │   Initiative   │         │   Assignment     │
    └───────────────┘           └────────────────┘         └────────┬─────────┘
                                                                    │
                                                                    │ At resource
                                                                    │ create / update:
                                                                    │ ARM evaluates
                                                                    ▼
                                                          ┌──────────────────┐
                                                          │  Effect fires    │
                                                          │  ── Audit        │
                                                          │  ── Deny         │
                                                          │  ── Append       │
                                                          │  ── Modify       │
                                                          │  ── DeployIfNot… │
                                                          │  ── AuditIfNot…  │
                                                          └──────────────────┘

    Assignments INHERIT DOWN — a policy at MG scope cascades to all subs / RGs / resources.
```

## Policy vs RBAC

| RBAC                             | Azure Policy                        |
| -------------------------------- | ----------------------------------- |
| Controls **who** can do actions  | Controls **what** configurations are allowed |
| Identity-based                   | Resource-configuration-based        |
| Answers: "Can this user create a VM?"  | Answers: "If a VM is created, does it meet standards?" |
| Assigned to identity + scope     | Assigned to scope only              |
| Complementary — use **both**     |                                     |

> **Exam-hot** — A user with Contributor RBAC can create a VM. If Policy says only certain regions are allowed, Policy will *deny* the creation despite the RBAC allowing it. Both must pass.

## Resource locks

Prevent accidental deletion or modification of critical resources.

### The two lock types

| Lock             | Modify allowed? | Delete allowed? |
| ---------------- | :-------------: | :-------------: |
| **ReadOnly**     | No              | No              |
| **Delete / CanNotDelete** | Yes    | No              |

- Applied at **subscription**, **resource group**, or **individual resource** scope
- **Inherit down** the hierarchy — a lock on the RG applies to every resource in it
- Applied on top of RBAC — even Owners are affected
- To modify a locked resource, first remove the lock (requires `Microsoft.Authorization/*` permissions)

**Common uses**
- Protect production databases from accidental delete
- Lock a shared network resource that must never be modified

> **Gotcha** — ReadOnly can have unintended side effects. For a VM with ReadOnly, you can't even start / stop it because those actions modify state. Test carefully.

## Service Trust Portal

Microsoft's public repository at **`servicetrust.microsoft.com`** for compliance documentation.

### Contents

- Independent audit reports (SOC 1 / 2 / 3, ISO 27001 / 27017 / 27018, FedRAMP, HIPAA/HITRUST)
- Compliance guides (industry- and region-specific)
- White papers
- FAQs and support docs
- **Microsoft Purview Compliance Manager** — assess and manage your compliance posture

### When to use

- Auditor asks for Microsoft's SOC 2 Type II report → download it here
- Preparing a security questionnaire — pull audit letters and attestations
- Verifying that a specific Azure service is in-scope for a compliance framework

## Governance stack — how everything fits

```
Management Groups
   │  ← Apply Policy assignments + RBAC + Locks at this scope
   ▼
Subscriptions
   │  ← Same
   ▼
Resource Groups
   │  ← Same
   ▼
Resources
   │  ← Lowest scope for locks and RBAC
   ▼
Data inside resources  ← governed by Purview
```

Policies and RBAC evaluated at deployment / management time by ARM.
Purview is separate — it discovers and governs the *data* inside those resources.

---

## Gotchas — governance & compliance

- **Purview ≠ Policy.** Purview = data. Policy = resource config.
- **RBAC and Policy are complementary**, not alternatives.
- **Locks apply to Owners too.** No one can delete a lock-protected resource until the lock is removed.
- **ReadOnly can block routine operations** (start/stop VM). Use with care.
- **Service Trust Portal is a document repository**, not a live compliance tool. For live posture, use Microsoft Purview Compliance Manager.
- **Policy scope inherits down.** A policy set on a management group cascades to every subscription, RG, and resource below it.

---

## Anchor facts

- **Purview** — data governance across your estate.
- **Azure Policy** — enforces resource configuration standards. Effects: audit, deny, append, modify, deployIfNotExists, auditIfNotExists, disabled.
- **Resource locks** — ReadOnly (no modify, no delete), Delete (no delete, modify OK). Inherit down.
- **Service Trust Portal** — public documentation of Microsoft's compliance certifications.
- **RBAC + Policy = complete control.** RBAC = who. Policy = what.
- Both are enforced by ARM at request time — both must pass.
