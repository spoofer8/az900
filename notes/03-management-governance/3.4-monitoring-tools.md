# 3.4 · Describe monitoring tools in Azure

**Domain 3 · Azure management and governance** — 30–35% of exam

Three tools cover this objective at fundamentals level: **Azure Advisor** (recommendations), **Azure Service Health** (Azure-side incidents), and **Azure Monitor** (metrics, logs, alerts, insights). Know each one's *purpose* and the boundary between them.

---

## Azure Advisor

Personalised, best-practice recommendations for your Azure environment.

### The five Advisor pillars

| Pillar                     | Examples of recommendations                                            |
| -------------------------- | ---------------------------------------------------------------------- |
| **Reliability**            | Enable backup, use availability zones, configure DR                    |
| **Security**               | Enable MFA, restrict inbound ports, enable Defender for Cloud plans    |
| **Performance**            | Right-size VMs, upgrade storage tiers, enable accelerated networking   |
| **Cost**                   | Right-size or shut down idle VMs, buy reservations, delete unused resources |
| **Operational excellence** | Tag resources, monitor with Azure Monitor, enable diagnostics          |

- Recommendations are prioritised (High / Medium / Low impact)
- Free — surfaced on the Advisor blade in the portal
- Some Security recommendations link into **Microsoft Defender for Cloud** and feed **Secure Score**

> **Exam-hot** — Memorise the five pillars in order. "Which Advisor category?" questions are common.

## Azure Service Health

Personalised view of Azure's own health, filtered to what affects **your resources**.

### The four content types

| Category                | Content                                                                          |
| ----------------------- | -------------------------------------------------------------------------------- |
| **Service issues**      | Ongoing Azure incidents impacting your services                                  |
| **Planned maintenance** | Upcoming maintenance events (VM reboots, region maintenance) with impact windows |
| **Health advisories**   | Advisory notices, deprecations, service changes                                  |
| **Security advisories** | Security-related notices, vulnerabilities requiring your action                  |

### Service Health vs Resource Health

| Service Health                                     | Resource Health                                    |
| -------------------------------------------------- | -------------------------------------------------- |
| Azure-side issues affecting **your** services      | Health of an **individual resource** you own       |
| "Azure has an ongoing incident in West Europe"     | "This specific VM is currently unavailable"        |
| Configurable alerts to notify teams                | Automatic diagnostics                              |

### Azure Status page

Public dashboard at **`status.azure.com`** — worldwide Azure health at a glance, not filtered to you.

> **Gotcha** — Three related-but-different tools:
> - **Azure Status** = public global page (everyone)
> - **Service Health** = filtered to your subscriptions
> - **Resource Health** = specific to your individual resources

### Three health surfaces — different audiences

```
   status.azure.com          Azure Service Health          Resource Health
   ────────────────          ────────────────────          ───────────────
   Public dashboard          Filtered to YOUR              Per-resource
   Everyone sees it          subscriptions                 status (this VM,
                             (personalised)                this DB, this SA)

   "Is Azure globally OK?"   "Is Azure having an           "Is my particular
                              incident that affects        VM up right now?"
                              MY services?"
```

## Azure Monitor

The **umbrella** for all Azure observability — metrics, logs, alerts, dashboards, insights.

### From source to action — the data path

```
   SOURCES                        STORES / ENGINES              CONSUMERS
   ───────                        ────────────────              ─────────
   Azure resources  ─metrics──▶  ┌────────────────┐   ────▶  Metrics Explorer
   Guest OS / apps  ─logs──────▶ │  Log Analytics │   ────▶  Workbooks / dashboards
   Activity log     ─events────▶ │    workspace   │   ────▶  App Insights views
   App Insights SDK ─traces────▶ │  (KQL-queried) │   ────▶  Sentinel · Defender
                                 └────────┬───────┘
                                          │  signal + condition
                                          ▼
                                 ┌────────────────────┐
                                 │    Alert rule      │  metric · log ·
                                 │  (fires condition) │  activity · smart
                                 └────────┬───────────┘
                                          │  fires
                                          ▼
                                 ┌────────────────────┐
                                 │   Action Group     │  email · SMS · voice
                                 │                    │  webhook · Logic App
                                 │                    │  Function · Runbook
                                 └────────────────────┘

    App Insights data lives in a Log Analytics workspace under the hood.
    Alerts don't act by themselves — they fire THROUGH an Action Group.
```

### Data ingested by Azure Monitor

| Data type      | Source examples                                                                     |
| -------------- | ----------------------------------------------------------------------------------- |
| **Metrics**    | Numerical, near-real-time (CPU %, memory, request rate)                             |
| **Logs**       | Text records: activity, resource logs, VM guest OS logs, application traces         |
| **Traces**    | Distributed request traces from Application Insights                                |
| **Changes**    | Change history for resources                                                        |

### Key components

| Component                      | What it does                                                                        |
| ------------------------------ | ----------------------------------------------------------------------------------- |
| **Metrics Explorer**           | Chart and analyse numerical metrics                                                 |
| **Log Analytics workspace**    | Central store for logs; queryable via **KQL** (Kusto Query Language)                |
| **Alerts**                     | Trigger on metric / log / activity conditions; fire an Action Group                 |
| **Action Groups**              | Notification / automation targets (email, SMS, voice, webhook, Logic App, Function, Runbook) |
| **Autoscale**                  | Metric-driven scale-out/scale-in for VMSS, App Service, more                        |
| **Application Insights**       | Application performance monitoring (APM) for web / API apps                         |
| **Container Insights**         | Monitor AKS clusters, container performance                                         |
| **VM Insights**                | OS-level performance and dependency mapping for VMs                                 |
| **Network Watcher**            | Network monitoring, packet capture, connection troubleshooting                      |
| **Workbooks & dashboards**     | Custom interactive reports built on Monitor data                                    |

### Log Analytics + KQL

- **Log Analytics workspace** is where log data is stored and queried
- Query with **Kusto Query Language (KQL)**, similar to SQL
- Example:
  ```kql
  Perf
  | where TimeGenerated > ago(1h)
  | where CounterName == "% Processor Time"
  | summarize avg(CounterValue) by Computer
  ```
- Same engine underlies **Sentinel** and **Defender for Cloud**

### Application Insights

- APM for web apps, APIs, mobile back-ends
- Captures:
  - Request telemetry (rate, duration, failure)
  - Dependency calls (SQL, HTTP, storage)
  - Exceptions (with stack traces)
  - Page views + client-side JS errors (with SDK)
  - Custom events and metrics
- Live Metrics stream, application map, end-to-end transaction diagnostics
- Now part of Azure Monitor and stored in a Log Analytics workspace

### Alerts — the flow

```
1. Signal source           (metric | log | activity log | Service Health)
        ↓
2. Alert rule condition    (e.g. CPU > 90% for 5 min)
        ↓
3. Alert fires
        ↓
4. Action Group            (email → SMS → Function → Webhook → Runbook)
```

Signal types:
- **Metric alerts** — near-real-time (~ 1 min)
- **Log alerts** — periodic queries against Log Analytics
- **Activity log alerts** — subscription-level events
- **Smart detection** — AI-driven anomaly detection (App Insights)

## Bringing it together — which tool for what?

| Scenario                                                          | Tool                          |
| ----------------------------------------------------------------- | ----------------------------- |
| "Is Azure having an outage in West US right now?"                 | Azure Service Health          |
| "Is my specific VM in West US currently up?"                      | Resource Health               |
| "What's my subscription's cost / performance / security posture?" | Azure Advisor                 |
| "What's the CPU of my VMs over the last 24 h?"                    | Azure Monitor — Metrics       |
| "Search the last week of firewall logs for a specific IP"         | Azure Monitor — Log Analytics |
| "Track failed HTTP requests to my web app"                        | Application Insights          |
| "Auto-scale my VMSS when CPU is high"                             | Monitor Autoscale             |
| "Notify me when a resource is created outside allowed regions"    | Monitor Alerts + Activity Log |

## Gotchas — monitoring

- **Advisor is not a monitoring tool** — it's a recommendation engine. It reads *from* Monitor data.
- **Service Health ≠ Resource Health.** Provider-side incident vs your-resource-specific status.
- **App Insights is part of Azure Monitor now** — data is stored in a Log Analytics workspace under the hood.
- **KQL is not SQL** but resembles it; every log-based Monitor query uses KQL.
- **Alerts don't act by themselves** — they fire *through* an Action Group.
- **Container Insights, VM Insights, Network Watcher** are all Monitor features, not separate products.

---

## Anchor facts

- **Advisor pillars: reliability · security · performance · cost · operational excellence.**
- **Service Health**: Azure incidents; four categories (issues, planned maintenance, health advisories, security advisories).
- **Resource Health**: individual resource up/down status.
- **Azure Monitor** = umbrella: **Metrics + Logs + Alerts + App Insights + Container/VM Insights + Network Watcher.**
- **Log Analytics** stores logs; **KQL** queries them.
- **Alerts fire → Action Groups notify or automate.**
