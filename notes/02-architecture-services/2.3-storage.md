# 2.3 · Describe Azure storage services

**Domain 2 · Azure architecture and services** — 35–40% of exam

Two acronym clusters you must fully own: **redundancy** (LRS, ZRS, GRS, GZRS + read-access variants) and **access tiers** (Hot, Cool, Cold, Archive). Expect at least two exam questions on each.

---

## The Azure Storage account

A storage account is the container for Azure's storage services. One account can host multiple services simultaneously.

**Properties chosen at creation** (some can't be changed later):
- Name (globally unique, 3–24 lowercase alphanumeric)
- Region
- Performance tier: **Standard** or **Premium**
- Redundancy option (see below)
- Access tier default: Hot / Cool / Cold (Blob only)

> **Anchor** — Azure Storage provides **99.999999999%** (eleven-nines) durability of objects per year regardless of redundancy option. Redundancy affects *availability* and *disaster tolerance*; durability is a floor across the board.

### Data shape → service

Three data shapes map to different Azure surfaces.

| Data shape          | Meaning                                    | Azure surfaces                                     |
| ------------------- | ------------------------------------------ | -------------------------------------------------- |
| **Structured**      | Strict schema, rows and columns            | Azure SQL DB, Azure Files, Managed Disks (block-level access) |
| **Semi-structured** | Loose schema, key/value or document        | Azure Cosmos DB, Azure Table Storage               |
| **Unstructured**    | Any bytes — video, image, log, backup      | Azure Blob Storage, Azure Data Lake Storage Gen2   |

> Blob = **unstructured**. Table = **semi-structured**. SQL / Files / Disks = **structured** surfaces.

## The five storage-service types

| Service        | What it stores               | Access via                          | Typical use                                             |
| -------------- | ---------------------------- | ----------------------------------- | ------------------------------------------------------- |
| **Blob**       | Unstructured objects         | REST / SDK                          | Images, video, logs, backups, big-data lakes            |
| **File**       | SMB / NFS file shares        | SMB 2.1/3.0, NFS 4.1, REST          | Lift-and-shift file shares, shared config, home dirs    |
| **Queue**      | Messages (up to 64 KB each)  | REST / SDK                          | Decouple app components, background job queues          |
| **Table**      | Key-value NoSQL              | REST / SDK                          | Cheap NoSQL (largely superseded by Cosmos DB Table API) |
| **Disk**       | VM block storage             | Attached to a VM as a virtual disk  | OS + data disks for VMs                                 |

### Blob types

| Blob type   | Purpose                                                       |
| ----------- | ------------------------------------------------------------- |
| **Block**   | Text and binary data; the default (documents, images, videos) |
| **Append**  | Optimised for append-only workloads (log files)               |
| **Page**    | Random-access reads/writes; used by managed disks             |

## Redundancy — the 6 options

Two axes: **where copies live** (single DC vs zone vs region) and whether the secondary is **read-accessible**.

| Option        | Copies | Where                                | Protects against                                    |
| ------------- | :----: | ------------------------------------ | --------------------------------------------------- |
| **LRS**       | 3      | Single datacentre                    | Disk / server / rack failure                        |
| **ZRS**       | 3      | Across 3 availability zones          | LRS + single-DC / zone failure                      |
| **GRS**       | 6      | 3 in primary DC + 3 in paired region | Regional disaster                                   |
| **GZRS**      | 6      | 3 zones in primary + 3 in secondary  | Regional disaster + zonal in primary                |
| **RA-GRS**    | 6      | Same as GRS, secondary is *readable* | Same as GRS + provides read access to secondary     |
| **RA-GZRS**   | 6      | Same as GZRS, secondary is *readable* | Same as GZRS + read access to secondary            |

**Mnemonic:**
- **L**ocally: 1 DC
- **Z**one: 1 region, 3 zones
- **G**eo: 2 regions
- **GZ**: geo + zone in primary
- **RA-**: adds read access to secondary

### Where the copies live

```
   LRS       [ ● ● ● ]  single datacentre                           3 copies
                        protects: disk / server / rack failure

   ZRS       [ ● ][ ● ][ ● ]  three AZs in one region               3 copies
             AZ1  AZ2  AZ3    protects: single-DC / zone outage

   GRS       [ ● ● ● ]  ═══ async ═══▶  [ ● ● ● ]                   6 copies
             primary DC                  paired region              secondary:
                                                                    NOT readable

   GZRS      [ ● ][ ● ][ ● ] ═ async ═▶ [ ● ● ● ]                   6 copies
             AZ1  AZ2  AZ3                paired region              secondary:
             (ZRS in primary)                                       NOT readable

   RA-GRS    same as GRS   +  SECONDARY IS READABLE                 read replica
   RA-GZRS   same as GZRS  +  SECONDARY IS READABLE                 for disaster
                                                                     reads / DR tests

              L → Z → G → GZ         = wider blast radius protected
              RA- prefix             = adds read access to the secondary
```

> **Exam-hot** — Failover from primary to secondary in GRS/GZRS can be manual or provider-initiated. Without `RA-`, you can't read from the secondary until failover completes.

## Access tiers (Blob only)

| Tier         | Storage $ | Access $ | Retrieval latency | Min retention | Best for                                    |
| ------------ | :-------: | :------: | :---------------: | :-----------: | ------------------------------------------- |
| **Hot**      | Highest   | Lowest   | Milliseconds      | None          | Frequently accessed data                    |
| **Cool**     | Lower     | Higher   | Milliseconds      | 30 days       | Infrequently accessed, kept ≥ 30 days       |
| **Cold**     | Lower still | Higher | Milliseconds      | 90 days       | Rarely accessed, kept ≥ 90 days             |
| **Archive**  | Lowest    | Highest  | **Up to 15 hours** (rehydration) | 180 days | Long-term backup / archive               |

**Tier switching**
- Tiers can change at any time (via lifecycle management or manually)
- Early-deletion fees apply if you move data out of Cool/Cold/Archive before the min-retention period
- **Archive requires rehydration** to Hot/Cool before it can be read (up to 15 hours; can be requested "high priority" for faster)

## Encryption & security

- **All data encrypted at rest** by default with Microsoft-managed keys (AES-256)
- Optional **customer-managed keys (CMK)** via Key Vault
- **Encryption in transit** via HTTPS (enforce "secure transfer required")
- Access control:
  - Storage account keys (like passwords — rotate them!)
  - **Shared Access Signatures (SAS)** — time-limited, scoped access tokens
  - **Entra ID + RBAC** — preferred for user access
  - **Firewalls & VNet rules** — restrict source IPs / subnets
  - **Private endpoints** — private IP inside your VNet

## Moving data into Azure

### AzCopy
- Command-line tool for high-throughput copy
- Blob ↔ File ↔ local ↔ S3
- Great for scripted bulk uploads

### Azure Storage Explorer
- Cross-platform desktop GUI
- Manage blobs, files, queues, tables in a familiar tree view
- Convenient for ad-hoc uploads and permissions

### Azure File Sync
- Extend Windows Server file shares to Azure Files
- Local cache, tiering: hot data on-prem, cold in the cloud
- Multi-site sync

### Azure Migrate
- Discovery, assessment, and migration hub
- Handles servers (physical + virtual), databases (SQL Server), web apps, virtual desktops
- Free (you pay for the target Azure resources)

### Azure Data Box
- Physical devices Microsoft ships to you, you load with data, ship back
- Options:
  - **Data Box Disk** — up to 8 SSDs, ~35 TB total
  - **Data Box** — 100 TB rugged appliance
  - **Data Box Heavy** — 1 PB, wheeled
- For situations where the network is too slow to move the data in reasonable time
- **Data Box Gateway** and **Data Box Edge** are separate products for ongoing on-prem ↔ cloud data movement

### Summary — which tool?

| Scenario                                                    | Tool                    |
| ----------------------------------------------------------- | ----------------------- |
| Copy 10 GB of files from your laptop, quickly, scripted     | AzCopy                  |
| GUI to browse and manage blobs                              | Storage Explorer        |
| Extend on-prem Windows file shares to Azure                 | Azure File Sync         |
| Assess on-prem servers and plan a lift-and-shift            | Azure Migrate           |
| Ship 100 TB across a slow WAN                               | Data Box                |
| Continuous hybrid data movement                             | Data Box Gateway / Edge |

## Gotchas — storage

- **Storage account names are globally unique** across all of Azure, not just your subscription.
- **Redundancy is set per account** (though some services within can have different options).
- **Archive tier is *not* immediately readable** — plan for rehydration time.
- **Egress costs money** — GRS replication is free, but reading from a secondary region *may* incur cross-region egress.
- **Data Box vs Data Box Gateway** — Box is one-time bulk transfer; Gateway is ongoing.
- **ZRS ≠ GRS** — one is zonal (in-region), the other is geographical (cross-region).

---

## Anchor facts

- Services: **Blob, File, Queue, Table, Disk.**
- Redundancy: **LRS (3, 1 DC), ZRS (3 zones), GRS (6, 2 regions), GZRS (6, ZRS+geo), RA-GRS, RA-GZRS.**
- Tiers: **Hot > Cool > Cold > Archive.** Archive rehydration up to 15 hours.
- All data encrypted at rest by default (AES-256).
- Migration ladder: **AzCopy → Storage Explorer → File Sync → Migrate → Data Box.**
