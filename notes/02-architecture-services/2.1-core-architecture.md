# 2.1 · Describe the core architectural components of Azure

**Domain 2 · Azure architecture and services** — 35–40% of exam

The most heavily tested topic in the entire exam. Every question here is worth defending.

---

## Azure's physical hierarchy

```
Geography  →  Region  →  Availability Zone  →  Datacentre
```

| Layer                | Definition                                                                                                     |
| -------------------- | -------------------------------------------------------------------------------------------------------------- |
| **Geography**        | A discrete market with data-residency & compliance boundaries (e.g. United States, Europe, Asia Pacific)       |
| **Region**           | A set of datacentres within a latency-defined perimeter, connected by a dedicated regional low-latency network |
| **Availability Zone**| One or more datacentres with independent power, cooling, and networking within a region                        |
| **Datacentre**       | A physical facility containing servers                                                                         |

---

## Regions

- **60+ Azure regions** globally — more than any other cloud provider
- Every region belongs to a geography
- Each region has one or more datacentres
- **Region latency** — services in the same region talk quickly; cross-region talk is slower and often has egress cost

### Special region types

| Type                                    | Purpose                                                                             |
| --------------------------------------- | ----------------------------------------------------------------------------------- |
| **Standard**                            | The typical multi-tenant public Azure region                                        |
| **Sovereign / dedicated**               | Isolated instances of Azure for specific compliance or trust needs                  |
|  ↳ Azure Government                     | US federal, state, local government + contractors (physically isolated in the US)   |
|  ↳ Azure China (operated by 21Vianet)   | Physically isolated instance of Azure in China, operated by 21Vianet, not Microsoft |

> **Gotcha** — Sovereign regions are **not** just regions with restricted access. They are separate instances of Azure with their own portals, sign-in URLs, and (in China's case) a different operator.

---

## Region pairs

- Every Azure region is paired with another region within the same geography
- **Typically 300+ miles apart** for disaster isolation
- Provides:
  - Automatic **geo-replication** for services like GRS storage
  - **Sequential updates** — Azure never updates both paired regions at the same time
  - Priority for region recovery during a broad outage
- Examples: East US ↔ West US · North Europe ↔ West Europe · UK South ↔ UK West
- **Brazil South** is paired with **South Central US** — cross-geography exception

> **Exam-hot** — Region pairs are used for *disaster recovery*, not for regular high availability. HA uses availability zones.

### Region pair — the shape

```
                         Geography (e.g. Europe)
   ┌───────────────────────────────────────────────────────────────────────┐
   │                                                                       │
   │    Region A                    ~300+ miles                Region B    │
   │  (North Europe)  ═══════════════════════════════════▶  (West Europe)  │
   │                                                                       │
   │  ┌───┐┌───┐┌───┐                                       ┌───┐┌───┐┌───┐│
   │  │AZ1││AZ2││AZ3│  ── GRS auto-replication data ────▶   │AZ1││AZ2││AZ3││
   │  └───┘└───┘└───┘                                       └───┘└───┘└───┘│
   │                                                                       │
   │  Sequential Azure updates never touch both regions at the same time   │
   │  Region B gets priority for recovery during a broad outage            │
   │                                                                       │
   └───────────────────────────────────────────────────────────────────────┘

     Region pair  = disaster-recovery boundary (cross-region)
     Availability zone = high-availability boundary (in-region)
```

## Availability Zones (AZs)

- **Minimum three physically separate zones** per zone-enabled region
- Each zone = one or more datacentres with **independent power, cooling, and networking**
- Zones connected by high-speed private fibre
- Not every region supports AZs (check the docs for the specific region)

### AZ layout inside a region

```
                    ┌───────────── Region ─────────────┐
                    │                                  │
                    │   AZ 1         AZ 2         AZ 3 │
                    │  ┌─────┐     ┌─────┐     ┌─────┐ │
                    │  │  DC │     │  DC │     │  DC │ │
                    │  │─────│     │─────│     │─────│ │
                    │  │power│     │power│     │power│ │  independent power
                    │  │cool │     │cool │     │cool │ │  independent cooling
                    │  │netwk│     │netwk│     │netwk│ │  independent network
                    │  └──┬──┘     └──┬──┘     └──┬──┘ │
                    │     └───────────┼───────────┘    │
                    │       high-speed private fibre   │
                    └──────────────────────────────────┘

              Spread VMs across AZs → 99.99% SLA
              A DC outage takes down one zone, not the region
```

### What AZs protect against

| Failure                    | AZ-redundant deployment protects? |
| -------------------------- | :-------------------------------: |
| Single VM crash            | Yes (if load-balanced)            |
| Rack / hardware fault      | Yes                               |
| Datacentre outage          | Yes                               |
| Network split in the DC    | Yes                               |
| Region-wide outage         | No — use region pairs / GRS       |
| Meteor strike on geography | No                                |

### AZ vs Availability Set

| Feature       | Availability Zone                | Availability Set                        |
| ------------- | -------------------------------- | --------------------------------------- |
| Scope         | Across multiple datacentres      | Within one datacentre                   |
| Protects from | Datacentre / power / network     | Rack / update / power at rack level     |
| SLA (VM)      | 99.99%                           | 99.95%                                  |
| Concept       | Zone (1, 2, 3)                   | Fault domain + update domain            |

> **Gotcha** — "Availability zone" and "availability set" are near-synonyms that mean very different things. AZ = across datacentres. Set = within a datacentre.

## Azure resources

Any single manageable item in Azure: a VM, a storage account, a SQL database, a virtual network. Each resource:

- Has a globally unique resource ID
- Belongs to exactly **one resource group**
- Lives in exactly **one region** (some are global — e.g. Front Door, Traffic Manager)
- Is created and managed through Azure Resource Manager (ARM)

---

## Resource groups (RGs)

A logical container for a set of related resources.

**Rules**
- Every resource lives in exactly one RG
- A resource can be moved between RGs (with limits)
- RGs themselves live in a subscription
- RGs have a region (metadata location) — but they can contain resources in *other* regions
- Delete an RG → all its resources are deleted (RG = lifecycle boundary)
- RG has a scope for RBAC, policy, and locks

**Good uses for an RG**
- Group resources by application, lifecycle, environment (dev / test / prod), or department

---

## Subscriptions

A billing and access-management container that holds resource groups.

**Rules**
- Every resource group lives in one subscription
- A subscription has a single billing account and account owner
- A subscription is a security / policy / quota boundary
- A single organisation typically has **multiple** subscriptions (e.g. dev, test, prod, per-department)
- You can transfer a subscription between billing accounts

**Common patterns**
- Environment isolation (dev / prod)
- Cost allocation
- Quota headroom (each subscription has service limits)
- Compliance boundary

---

## Management groups

Containers **above** subscriptions, for scaled governance across many subscriptions.

- Support up to **six levels of hierarchy** below the root
- One tenant has exactly one root management group
- A subscription can be in exactly one management group
- **RBAC roles and Azure Policies** applied at a management group cascade down to all child subs / RGs / resources

### The full hierarchy

```
Tenant (Microsoft Entra ID)
   └── Root Management Group
        └── Management Groups (nested up to 6 deep)
             └── Subscriptions
                  └── Resource Groups
                       └── Resources
```

> **Exam-hot** — Learn this hierarchy cold. Questions often ask "at what scope can this policy / RBAC / lock be applied?"

---

## Azure Resource Manager (ARM)

The unified deployment and management layer for every Azure resource.

- Every request (Portal, CLI, PowerShell, SDK, REST) goes through ARM
- ARM handles authentication (with Entra ID), authorisation (RBAC), and orchestration
- Enables:
  - **Idempotent deployments** (ARM templates, Bicep)
  - **Tagging**
  - **Locking**
  - **Policy enforcement**
  - **RBAC**

---

## Gotcha checklist

- **Region pair vs AZ** — pair is for DR (cross-region), AZ is for HA (in-region).
- **A resource has a region; an RG has a region — they don't have to match.**
- **Delete an RG → everything in it is gone.** No soft-delete for the RG itself.
- **You can nest management groups up to 6 levels**, but only *one* root.
- **Sovereign regions are separate instances of Azure**, not just regions with locked access.
- **Some services are global** (Entra ID, Front Door, Traffic Manager, DNS) — they don't have a region.

---

## Anchor facts to memorise

- 60+ regions, min 3 AZs per zone-enabled region, region pairs ~300+ miles apart.
- Hierarchy: Tenant → Root MG → MGs → Subs → RGs → Resources.
- Region pair = DR. AZ = HA. Availability Set = rack-level HA in one DC.
- ARM is *the* control plane. Everything goes through it.
- SLA ladder: 99.9% (single) → 99.95% (Availability Set) → 99.99% (AZ).
