# 2.4 · Describe Azure identity, access, and security

**Domain 2 · Azure architecture and services** — 35–40% of exam

Wide topic. Microsoft renamed *Azure Active Directory* to **Microsoft Entra ID** — the exam uses the new name. Older content may still say Azure AD.

---

## Microsoft Entra ID — the identity foundation

- Microsoft's cloud-based **identity and access management** service
- Formerly known as **Azure Active Directory**
- **Shared identity plane** for Azure, Microsoft 365 / Office 365, Dynamics 365, and Microsoft consumer services (Outlook.com, Xbox, Skype) — **not Azure-only**
- **Not** the same as on-prem Active Directory Domain Services — Entra ID does not do LDAP, NTLM, Kerberos, or Group Policy
- Provides: authentication, SSO to thousands of SaaS apps, MFA, Conditional Access, external identities, device management (with Intune)

### Entra ID vs Windows Server AD DS

| Feature                       | Entra ID                          | Windows Server AD DS         |
| ----------------------------- | --------------------------------- | ---------------------------- |
| Protocols                     | OAuth 2.0, OpenID Connect, SAML   | LDAP, Kerberos, NTLM         |
| Structure                     | Flat (tenants, groups)            | Hierarchical (OUs, forests)  |
| Group Policy                  | No (use Intune)                   | Yes                          |
| Cloud native                  | Yes                               | No                           |
| Manages Azure resource access | Yes                               | No (by itself)               |

### Microsoft Entra Domain Services

Managed **AD DS-compatible** service — provides Kerberos, NTLM, LDAP, Group Policy inside Azure without deploying domain controllers. Useful for legacy apps that need AD DS but don't want on-prem infrastructure.

## Authentication vs authorisation

| Term              | Question it answers          | Example                                    |
| ----------------- | ---------------------------- | ------------------------------------------ |
| **Authentication** | Who are you?                 | Verifying username + password + MFA        |
| **Authorisation**  | What are you allowed to do?  | Checking group membership before granting access |

> **Exam-hot** — Read every scenario for this distinction. Signing in = authn. Deciding what you can then click on = authz.

### The sign-in flow — two distinct checks

```
     User ──▶ [ Password + MFA prompt ] ──▶  Entra ID          (1) AUTHENTICATION
                                             verifies              "who are you?"
                                             the identity              │
                                                                       │
                                                                       ▼
                                             RBAC lookup          (2) AUTHORISATION
                                             (role + scope        "what can you do?"
                                              for the target
                                              resource)                │
                                                                       │
                                                                       ▼
                                             Grant or deny the action on the resource
                                             ▲
                                             │
                                             Conditional Access may block or add
                                             extra requirements (MFA, compliant device…)
                                             at this step in real-time
```

## Authentication methods

### Passwords
- Least secure; subject to phishing, guessing, credential stuffing
- Should be paired with MFA

### Multi-factor authentication (MFA)
Uses **two or more** of:

| Factor              | Examples                                                    |
| ------------------- | ----------------------------------------------------------- |
| Something you know  | Password, PIN, security question                            |
| Something you have  | Phone (SMS / app), FIDO2 key, smart card                    |
| Something you are   | Fingerprint, face, iris (biometrics)                        |
| Somewhere you are   | Location, IP range, GPS — evaluated by Conditional Access   |

The first three factors are the classic set; **"somewhere you are"** shows up on the exam as the fourth category (Microsoft lists it alongside know / have / are). Conditional Access uses it to require MFA when a sign-in comes from an unusual location.

MFA is included with Entra ID Free (for admins via Security Defaults) and full-fledged in Entra ID P1/P2.

### Passwordless
Removes the password entirely. Options:

- **Windows Hello for Business** — biometric or PIN, TPM-bound
- **Microsoft Authenticator app** — approve sign-in on your phone
- **FIDO2 security keys** — hardware key, phishing-resistant

### Single sign-on (SSO)
- One authentication session gives access to multiple apps
- Reduces credential fatigue and password reuse
- Enabled through Entra ID for federated apps

## External identities

Entra ID supports collaboration and consumer scenarios.

| Feature                    | Purpose                                                                  |
| -------------------------- | ------------------------------------------------------------------------ |
| **Entra External ID for B2B** | Invite external users (partners, contractors) as guest users. They keep their own identity, you grant access to your resources. |
| **Entra External ID for B2C** | Provide identity for your customer-facing apps. Users sign up with email, social accounts (Google, Facebook), or local accounts. |

## Conditional Access

Policy engine that evaluates **signals** and applies **controls** at sign-in.

**Signals (inputs)**
- User / group
- Application being accessed
- Device state (compliant, hybrid-joined)
- Location (IP range, country)
- Real-time risk (sign-in risk, user risk)

**Controls (outputs)**
- Grant access (with MFA, compliant device, etc.)
- Block access
- Require password change
- Session controls (limited experience, sign-in frequency)

**Requires** Entra ID P1 or higher. Free tier gets a simpler "Security Defaults" preset.

## Role-Based Access Control (RBAC)

Assign **who** (identity) has **what** access (role) to **which** resource (scope).

### The four fundamental built-in roles

| Role                       | Can do                                        |
| -------------------------- | --------------------------------------------- |
| **Owner**                  | Full access + manage access for others        |
| **Contributor**            | Full management access, **cannot** grant access to others |
| **Reader**                 | View resources; cannot change                 |
| **User Access Administrator** | Manage user access to Azure resources     |

Hundreds more built-in roles, plus custom roles for fine-grained control.

### Scope

RBAC assignments are made at a scope; permissions **inherit down** the hierarchy.

```
Management Group  ──▶ inherits to
   Subscription   ──▶ inherits to
      Resource Group  ──▶ inherits to
         Resource
```

Assign at the highest useful scope; over-permissioning at the top is a common mistake.

### RBAC vs Azure Policy

| RBAC                              | Azure Policy                              |
| --------------------------------- | ----------------------------------------- |
| Governs *who* can perform actions | Governs *what* configurations are allowed |
| Identity-based                    | Resource-based                            |
| Enforced at request time by ARM   | Enforced at request time by ARM           |
| Use both — they're complementary  |                                            |

## Zero Trust

Modern security model: **never trust, always verify.**

### The three principles

1. **Verify explicitly** — always authenticate and authorise based on all available signals (user, location, device, service, workload, data classification, anomalies)
2. **Use least-privilege access** — Just-In-Time (JIT), Just-Enough-Access (JEA), risk-based policies
3. **Assume breach** — segment access, verify end-to-end encryption, use analytics to improve defences

### Zero Trust — the three pillars

```
   ┌──────────────────────┐   ┌──────────────────────┐   ┌──────────────────────┐
   │  1. Verify           │   │  2. Least            │   │  3. Assume           │
   │     explicitly       │   │     privilege        │   │     breach           │
   ├──────────────────────┤   ├──────────────────────┤   ├──────────────────────┤
   │ Every request        │   │ Only the access      │   │ Segment, encrypt     │
   │ authenticated &      │   │ needed, only when    │   │ end-to-end, verify   │
   │ authorised on ALL    │   │ needed:              │   │ every session,       │
   │ signals — user,      │   │  • JIT (Just-In-     │   │ analyse everything   │
   │ device, location,    │   │    Time)             │   │ for anomalies.       │
   │ workload, data,      │   │  • JEA (Just-        │   │                      │
   │ real-time risk.      │   │    Enough-Access)    │   │ Detect fast,         │
   │                      │   │  • Risk-based        │   │ contain fast,        │
   │                      │   │    adaptive policy.  │   │ recover fast.        │
   └──────────────────────┘   └──────────────────────┘   └──────────────────────┘

     No implicit trust — anywhere. Every access is treated as if from an open network.
```

## Defense in depth

Layered security — no single control is your defence. The seven layers, from outside in:

| Layer                | Concerns                                                                 |
| -------------------- | ------------------------------------------------------------------------ |
| **Physical**         | Datacentre building security, badge access, cameras                      |
| **Identity & access** | Who you are, MFA, Conditional Access, RBAC                               |
| **Perimeter**        | DDoS Protection, Azure Firewall                                          |
| **Network**          | NSGs, ASGs, VNet segmentation                                            |
| **Compute**          | Endpoint protection, patch management, VM security                       |
| **Application**      | Secure coding, WAF, dependency scanning                                  |
| **Data**             | Encryption at rest and in transit, classification, DLP                   |

> **Exam-hot** — Know these seven layers, in order. Which control protects which layer?

## Security services

### Microsoft Defender for Cloud
- **CSPM** (Cloud Security Posture Management) + **CWPP** (Cloud Workload Protection Platform)
- Continuously assesses configuration against best-practice benchmarks
- Produces a **Secure Score** — your posture as a percentage; recommendations to raise it
- Enhanced security (paid) adds workload protection for servers, containers, databases, storage, key vault, etc.
- Covers Azure, AWS, GCP, and on-prem via Arc

### Microsoft Sentinel
- Cloud-native **SIEM** (Security Information & Event Management) and **SOAR** (Security Orchestration, Automation, and Response)
- Collects logs from Azure, other clouds, on-prem, third-party
- Detects threats using analytics + Microsoft threat intelligence
- Automates responses via playbooks (Logic Apps)

### Azure Key Vault
- Centralised store for **secrets** (passwords, connection strings), **keys** (encryption keys), and **certificates**
- Applications retrieve them at runtime with Entra ID identity — no secrets in code
- HSM-backed key protection available (Premium tier)

### Azure Firewall
- Managed, stateful, **L4–L7** firewall
- Threat intelligence-based filtering
- **Three rule kinds** —
  - **Network rules** — allow/deny by source/dest IP, port, protocol (TCP/UDP/ICMP)
  - **Application rules** — allow/deny by FQDN, incl. wildcards (`*.microsoft.com`); also supports FQDN tags (e.g. *WindowsUpdate*)
  - **NAT rules** — inbound DNAT translating a public IP to a backend resource
- Tiers: **Basic** (SMB), **Standard** (core features), **Premium** (adds **IDPS**, **TLS inspection**, **URL filtering**, **web categories**)

### Azure DDoS Protection
- **Basic** — always on for every Azure resource with a public IP; automatic mitigation of common attacks; no config required; no SLA credit
- **Standard / IP Protection Plans** — enhanced mitigation with:
  - **Adaptive tuning** — a machine-learning model profiles your traffic and adapts thresholds to your app's normal shape
  - **Attack analytics, alerts,** WAF integration
  - **Cost protection** — SLA credits for the scale-out charges incurred during a documented DDoS attack
  - Rapid Response support (Standard tier)

### NSG (Network Security Group)
- Rule set for allow/deny traffic to a subnet or NIC
- Stateful — return traffic automatically allowed
- Free

### Application Security Group (ASG)
- Group VMs by *application role* rather than IP address
- Use in NSG rules for cleaner, less brittle configs

## Gotchas — identity & security

- **Entra ID ≠ Windows Server AD.** Different protocols, different structure.
- **Owner vs Contributor** — Owner can grant access, Contributor cannot.
- **RBAC and Policy are complementary**, not alternatives. Use both.
- **DDoS Basic is on by default and free.** Standard is a paid, per-resource upgrade with SLA.
- **Sentinel = SIEM. Defender for Cloud = CSPM.** They complement each other; Sentinel ingests Defender's alerts.
- **Passwordless is *not* the same as MFA.** Passwordless replaces "something you know"; MFA still needs two factors of any type.

## Anchor facts

- Entra ID uses **OAuth 2.0, OpenID Connect, SAML** — not Kerberos/LDAP/NTLM.
- Zero Trust: **verify explicitly · least privilege · assume breach**.
- Defense in depth: **physical → identity → perimeter → network → compute → application → data.**
- RBAC fundamental roles: **Owner, Contributor, Reader, User Access Administrator.**
- Defender for Cloud → **Secure Score.** Sentinel → **SIEM/SOAR.**
- MFA needs two of: **know, have, are.**
