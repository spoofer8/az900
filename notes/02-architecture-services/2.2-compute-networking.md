# Part A · Compute

## Compute-service map

| Service                        | Type     | Use                                                                       |
| ------------------------------ | -------- | ------------------------------------------------------------------------- |
| **Virtual Machines**           | IaaS     | Full OS control, lift-and-shift, legacy workloads                         |
| **Virtual Machine Scale Sets** | IaaS     | Horizontally scale identical VMs, auto-heal, autoscale                    |
| **Azure App Service**          | PaaS     | Web apps, APIs, mobile back-ends. Multi-language                          |
| **Azure Container Instances**  | PaaS     | Single-container, serverless, per-second billing                          |
| **Azure Container Apps**       | PaaS     | Managed containers with KEDA-based autoscale                              |
| **Azure Kubernetes Service**   | PaaS     | Managed Kubernetes control plane; you manage node pools                   |
| **Azure Functions**            | Serverless | Event-driven code; pay per execution                                    |
| **Azure Virtual Desktop**      | PaaS     | Windows desktops delivered from Azure; multi-session Windows              |

### The compute ladder — control vs abstraction

```
   More control ◀────────────────────────────────────────▶ More abstraction
   Less abstraction                                        Less control

     VM ──▶ VMSS ──▶ AKS ──▶ App Service ──▶ Container Apps ──▶ Functions
    (IaaS)  (IaaS)  (PaaS)     (PaaS)        (PaaS/serverless)  (serverless)

   OS you  identical  managed   managed        managed containers   event-driven
   own     VMs auto-  K8s ctrl  runtime        + KEDA autoscale     pay per exec
           scaled     plane     stack          (scale to zero)      (scale to zero)

   ┌───────────────────────────┐               ┌───────────────────────────┐
   │  You manage more of the   │               │  Azure manages more of    │
   │  stack — patch OS, deps   │               │  the stack — you write    │
   │  Best for lift-and-shift  │               │  code only, no infra      │
   └───────────────────────────┘               └───────────────────────────┘
```

## Azure Virtual Machines (VMs)

- IaaS: you manage the OS, patching, apps, data
- Choose OS (Windows / Linux), size (SKU family: B, D, E, F, N, M, etc.), region, disk types
- Disks: **OS disk**, optional **data disks**, temporary local disk
  - Managed disks: LRS / ZRS options, HDD / SSD / Premium SSD / Ultra SSD
- **Stopped** vs **Stopped-deallocated**
  - Stopped: OS shut down, VM still allocated → still billed for compute
  - Stopped-deallocated: VM released from host → **no compute billing** (storage still billed)

### VM Availability options — the SLA ladder

| Setup                                        | SLA        |
| -------------------------------------------- | ---------- |
| Single VM with Premium SSD or Ultra Disk     | 99.9%      |
| Two or more VMs in an **Availability Set**   | 99.95%     |
| Two or more VMs across **Availability Zones**| 99.99%     |

---

## Virtual Machine Scale Sets (VMSS)

- Deploy and manage a set of identical, load-balanced VMs
- **Auto-scale** based on metrics (CPU, memory, custom)
- Auto-heal — replace failed instances
- Deploy across availability zones for HA
- Great for stateless workloads (web, APIs, workers)

---

## Availability Sets

- **Fault Domain (FD)** — a rack. Sharing an FD means sharing power / network. Default 3.
- **Update Domain (UD)** — a maintenance group. Only one UD reboots at a time. Default 5.
- VMs in a set are distributed across FDs and UDs automatically
- **Same datacentre.** For cross-datacentre resilience use Availability Zones.

---

## Azure App Service

- PaaS for hosting web apps, APIs, mobile back-ends
- Supports .NET, Java, Node.js, Python, PHP, Ruby, containers
- **App Service Plan** = the compute tier + region hosting your apps
- **Deployment slots** — staging environments with swap-to-production
- **Custom domains**, **TLS certs**, **auto-scale**, built-in CI/CD (GitHub, Azure Repos)

---

## Azure Container Instances (ACI)

- Simplest way to run a container in Azure — no orchestrator needed
- Per-second billing, fast start
- Good for: batch jobs, build agents, simple long-running services

## Azure Container Apps

- Managed container platform built on Kubernetes + KEDA + Dapr
- Serverless: scale to zero, scale on HTTP or event triggers
- Ideal for microservices without managing Kubernetes directly

## Azure Kubernetes Service (AKS)

- Managed **Kubernetes control plane** — free
- You manage node pools (Linux, Windows) — those are VMs you pay for
- Best when you need Kubernetes-native features (custom controllers, service mesh, complex networking)

---

## Azure Functions

- Serverless code execution — write a function, deploy it, Azure runs it
- Trigger types: HTTP, timer, queue, blob, Event Grid, Cosmos DB change, Service Bus
- Hosting plans:
  - **Consumption** — pay per execution, scale to zero
  - **Premium** — pre-warmed instances, VNet integration
  - **Dedicated (App Service Plan)** — run inside an existing App Service Plan

---

## Azure Virtual Desktop (AVD)

- Windows desktops and apps delivered from Azure
- **Multi-session Windows** — many users on one VM (Enterprise multi-session SKUs)
- Full user profile in Azure (FSLogix)
- Alternative to on-prem VDI; simpler licensing

# Part B · Networking

## The core building block: Virtual Network (VNet)

- A logically isolated IP address space in Azure — your private network in the cloud
- Regional: a VNet lives in one region
- Subdivided into **subnets**
- Subnets can host: VM NICs, private endpoints, gateways, App Service integration

---

## Subnets, NSGs, ASGs

| Concept                        | What it does                                                                       |
| ------------------------------ | ---------------------------------------------------------------------------------- |
| **Subnet**                     | A range within a VNet's address space                                              |
| **Network Security Group (NSG)** | Set of allow/deny rules; attach to a subnet or a NIC                             |
| **Application Security Group (ASG)** | Group VMs by application role for cleaner NSG rules                          |
| **Route table (UDR)**          | Custom next-hop routing overrides                                                  |
| **Service endpoint**           | Extend a VNet identity to certain Azure PaaS services over the Azure backbone      |
| **Private endpoint**           | Give a PaaS service a private IP inside your VNet                                  |

### A typical VNet topology

```
                          VNet   10.0.0.0/16   (one region)
     ┌────────────────────────────────────────────────────────────────────────┐
     │                                                                        │
     │   Subnet: web   10.0.1.0/24            Subnet: db    10.0.2.0/24       │
     │   ┌──────────────────────────┐         ┌──────────────────────────┐    │
     │   │  NSG: allow 80, 443 in   │         │  NSG: allow 1433 from    │    │
     │   │       from Internet      │         │       "web" subnet only  │    │
     │   │   ┌──┐   ┌──┐   ┌──┐     │  ────▶  │       ┌──┐               │    │
     │   │   │VM│   │VM│   │VM│     │         │       │DB│               │    │
     │   │   └──┘   └──┘   └──┘     │         │       └──┘               │    │
     │   └──────────────────────────┘         └──────────────────────────┘    │
     │                                                                        │
     └───────────────────────────────┬────────────────────────────────────────┘
                                     │  VNet peering
                                     ▼        (local peering = same region,
     ┌───────────────────────────────────────  global peering = across regions,
     │  Another VNet                            no transit through a 3rd VNet)
     └───────────────────────────────────────
```

## Connecting networks

### VNet peering
- Connect two VNets privately over the Microsoft backbone
- **Local peering** (same region), **Global peering** (across regions)
- No transit — traffic between two peered VNets is direct only, not through a third
- Low latency, high bandwidth

### VPN Gateway
- Encrypted tunnel over the **public internet**
- Types:
  - **Site-to-site (S2S)** — datacentre to Azure (needs an on-prem VPN device)
  - **Point-to-site (P2S)** — individual client to Azure
  - **VNet-to-VNet** — Azure-to-Azure via VPN

### ExpressRoute
- **Private** connection through a connectivity partner
- Does **not** traverse the public internet
- Higher bandwidth (up to 100 Gbps), lower latency, better SLA (99.95%)
- Used for hybrid cloud, migrations, low-latency links, regulated industries

| Feature       | VPN Gateway              | ExpressRoute                        |
| ------------- | ------------------------ | ----------------------------------- |
| Path          | Public internet, encrypted | Private, via partner              |
| Bandwidth     | Up to ~10 Gbps           | Up to 100 Gbps                      |
| Latency       | Variable                 | Low, predictable                    |
| Cost          | Lower                    | Higher                              |
| Setup time    | Minutes                  | Weeks (physical circuit)            |

### The two paths — visually

```
   On-prem                    Public internet                Azure region
   ┌────────┐  ═════════ VPN tunnel (IPsec, encrypted) ═════▶  ┌────────┐
   │  Your  │                                                  │  Your  │
   │   DC   │                                                  │  VNet  │
   │        │                                                  │        │
   │Router  │  ────── ExpressRoute (private circuit) ───────▶  │Gateway │
   └────────┘         via connectivity partner                 └────────┘
                      NOT on the public internet

   VPN Gateway: cheap, quick, over internet.
   ExpressRoute: fast, predictable, private, regulated-industry friendly.
```

## Load balancing services

Choose by OSI layer and by regional-vs-global scope.

| Service                     | OSI Layer | Scope    | Purpose                                                                    |
| --------------------------- | :-------: | -------- | -------------------------------------------------------------------------- |
| **Azure Load Balancer**     | L4        | Regional | TCP/UDP distribution across VMs                                            |
| **Application Gateway**     | L7        | Regional | HTTP(S) with URL routing, SSL termination, WAF (optional)                  |
| **Azure Front Door**        | L7        | Global   | Global HTTP(S) entry, CDN, WAF, path-based routing                         |
| **Traffic Manager**         | DNS       | Global   | DNS-based traffic routing (priority, weighted, performance, geographic)    |

**How to pick:**
- Same region, non-HTTP → **Load Balancer**
- Same region, HTTP → **Application Gateway**
- Multiple regions, HTTP → **Front Door**
- Multiple regions, non-HTTP or complex DNS routing → **Traffic Manager**

### The 2×2 you must memorise

```
                        │       REGIONAL              │       GLOBAL
     ───────────────────┼─────────────────────────────┼──────────────────────────
      L4  (TCP / UDP)   │   Azure Load Balancer       │   Traffic Manager
                        │   (fast, non-HTTP)          │   (DNS-based routing)
     ───────────────────┼─────────────────────────────┼──────────────────────────
      L7  (HTTP / S)    │   Application Gateway       │   Azure Front Door
                        │   (URL routing, WAF)        │   (CDN, WAF, path routing)
     ───────────────────┴─────────────────────────────┴──────────────────────────

     Front Door often sits IN FRONT of an Application Gateway.
     Traffic Manager returns a DNS answer — it doesn't proxy the traffic.
```

## Public vs private endpoints

| Concept                | Meaning                                                                                     |
| ---------------------- | ------------------------------------------------------------------------------------------- |
| **Public endpoint**    | A resource is reachable over the internet via a public IP or FQDN                           |
| **Service endpoint**   | A subnet's identity is presented to a PaaS service over the Azure backbone; still uses the public endpoint but skips public internet |
| **Private endpoint**   | A private IP inside your VNet for a specific instance of a PaaS service — most secure       |

## Gotchas — networking

- **VNets don't cross regions** — peer them or use Virtual WAN.
- **NSGs are stateful** — return traffic is automatically allowed.
- **VPN Gateway ≠ Application Gateway.** VPN Gateway is a *network gateway*; Application Gateway is a *load balancer*.
- **Front Door is global**; Application Gateway is regional. Front Door often *sits in front* of an Application Gateway.
- **Private endpoints** are per-instance (one storage account, one SQL DB); service endpoints are per-service-type.

## Anchor facts

- Compute: **VMs (IaaS), App Service (PaaS), ACI/Container Apps/AKS (containers), Functions (serverless), AVD (desktops).**
- VMSS = horizontal scale of VMs.
- VPN over internet; ExpressRoute over private circuit.
- Load Balancer L4, App Gateway L7 regional, Front Door L7 global, Traffic Manager DNS.
- VMs are IaaS. AKS control plane is PaaS. Functions are serverless.
