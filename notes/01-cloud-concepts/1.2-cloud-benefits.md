# 1.2 · Describe the benefits of using cloud services

**Domain 1 · Cloud concepts** — 25–30% of exam

The exam expects you to distinguish these terms *precisely*. Many questions test the boundary between two of them (e.g. reliability vs availability, scalability vs elasticity).

---

## The eight benefits — one line each

| Benefit             | One-line definition                                                                     |
| ------------------- | ---------------------------------------------------------------------------------------- |
| **High availability** | Systems remain up and accessible with minimal downtime                                 |
| **Reliability**       | Ability to recover from failures and continue functioning                              |
| **Scalability**       | Ability to increase or decrease capacity to meet demand                                |
| **Elasticity**        | *Automatic* scaling in response to real-time demand                                    |
| **Predictability**    | Consistent performance and cost — you know what you'll get and what you'll pay        |
| **Security**          | Provider-invested controls, tools, and compliance certifications                       |
| **Governance**        | Standardised control over what's deployed, how, and by whom                           |
| **Manageability**     | Central, template-driven control over resources and their configuration               |

---

## High availability

**Definition.** The percentage of time a system is operational and reachable, over a period.

- Achieved with redundancy (multiple copies of a resource across independent failure domains)
- Measured as a percentage — "four nines" = 99.99% ≈ 52 minutes downtime / year
- Delivered via **SLAs** (Service Level Agreements) — the provider's promise for a service
- Composite SLAs multiply: two services in series at 99.9% each ≈ 99.8% combined

> **Exam-hot** — VM SLA ladder:
> - Single VM (Premium SSD): **99.9%**
> - Availability Set: **99.95%**
> - Availability Zones: **99.99%**

### The "nines" ladder — SLA vs downtime

```
   SLA %        Yearly downtime      Deployment shape
   ─────        ──────────────       ─────────────────────────────────
   99%          ≈ 3.65 days          One VM, no redundancy
   99.9%        ≈ 8h 45m             Single VM, Premium SSD
   99.95%       ≈ 4h 22m             Availability Set (within one DC)
   99.99%       ≈ 52 min             Availability Zones (across DCs)
   99.999%      ≈ 5 min              Multi-region, active-active
   ↓                                 ↓
   fewer nines                       simpler / cheaper architecture
   more nines                        more redundancy, more cost
```

**Composite SLA — dependent services multiply**

```
   Web tier (99.9%)  ─▶  API tier (99.9%)  ─▶  DB tier (99.9%)
                                                    │
                            0.999 × 0.999 × 0.999 = 0.997
                                                    │
                                                    ▼
                          End-to-end SLA ≈ 99.7%   (worse than any single tier)
```

> **Gotcha** — **Preview and Free-tier services carry NO SLA (0%).** Every paid tier has a documented SLA, and higher pricing tiers generally come with higher SLAs. "Which of these has no SLA?" questions almost always point to a *Preview* feature or a *Free* tier.

## Reliability

**Definition.** The ability of a system to recover from failures and continue to function.

Availability answers *"is it up right now?"*. Reliability answers *"will it recover if something breaks?"*.

- Achieved through region pairs, geo-redundant storage, backups, disaster recovery
- Azure's global infrastructure (60+ regions, region pairs) is engineered for reliability
- **Azure Site Recovery** replicates workloads to a secondary region for DR

---

## Scalability

The ability to add or remove capacity to match demand. Two flavours:

| Type            | Meaning                                                     | Azure example                 |
| --------------- | ----------------------------------------------------------- | ----------------------------- |
| **Vertical**    | Change the *size* of one resource (bigger CPU, more RAM)    | Resize a VM to a larger SKU   |
| **Horizontal**  | Change the *number* of resources (more instances)           | Add VMs to a VM Scale Set     |

- Vertical scaling ("scale up / down") has a ceiling and often needs a restart
- Horizontal scaling ("scale out / in") is unlimited in principle and typically zero-downtime

### Vertical vs horizontal — the shape

```
   VERTICAL  (scale up ↑ / down ↓)          HORIZONTAL  (scale out → / in ←)
   ──────────────────────────────           ─────────────────────────────────

        ┌──────────────┐                          ┌──┐ ┌──┐ ┌──┐ ┌──┐
        │              │                          │VM│ │VM│ │VM│ │VM│  scale out
        │    BIG VM    │                          └──┘ └──┘ └──┘ └──┘  (add more)
        │              │
        │   ▓▓▓▓▓▓▓    │  scale up                     ┌──┐ ┌──┐
        │   ▓▓▓▓▓▓▓    │  (same VM,                    │VM│ │VM│  scale in
        │   ▓▓▓▓▓▓▓    │   more CPU/RAM)               └──┘ └──┘  (remove some)
        └──────────────┘

   Ceiling: max SKU Azure offers            Ceiling: near-unlimited
   Downtime: usually a restart              Downtime: typically zero
   Cost step: big jumps per SKU             Cost step: linear with instance count
```

## Elasticity

**Definition.** Automatic scaling that reacts to real-time demand — without human intervention.

- Elasticity is a *subset* of scalability, with automation added
- Azure Autoscale rules (metric-based) drive VMSS, App Service, and container platforms
- Serverless (Functions, Container Apps) is inherently elastic

> **Gotcha** — Scalability is the *capability*; elasticity is the *automatic* variant.

### Elasticity vs static capacity

```
   Load                        Demand  ▂▄▇█▇▄▂▁▂▅▇█▇▅▂▁▂▃▇█▇▃▂
    │
    │   Static capacity      ████████████████████████████████████  ← paid for peak
    │   (overprovisioned)         ← wasted →       ← wasted →         even when idle
    │
    │   Elastic capacity        ▂▄▇█▇▄▂▁▂▅▇█▇▅▂▁▂▃▇█▇▃▂            ← tracks demand,
    │   (autoscale)                                                    pays only for use
    └──────────────────────────────────────────────────────────────▶ Time

   Demand ─▶ Autoscale rules (CPU, queue depth, schedule) ─▶ VMSS / App Service / Functions
                                                              scales itself
```

## Predictability

Two aspects — both matter to enterprises.

| Aspect                    | What it means                                                             |
| ------------------------- | -------------------------------------------------------------------------- |
| **Performance predictability** | Consistent behaviour under load: capacity planning, autoscale, load balancing |
| **Cost predictability**        | Clear pricing tools: Pricing Calculator, TCO Calculator, Cost Management, budgets, alerts |

---

## Security

Cloud shifts *some* security to the provider, but not all. What Azure invests in on your behalf:

- Physical security of datacentres
- Network isolation between tenants
- Global network attack mitigation
- Compliance certifications (ISO, SOC, PCI, HIPAA, FedRAMP, GDPR)
- Encryption at rest by default for many services
- Identity platform (Microsoft Entra ID) with MFA, Conditional Access

You still own: data, identities, application-layer security, network configuration inside your VNet.

---

## Governance

**Definition.** Enforcing organisational standards and staying compliant.

Delivered via:

- **Azure Policy** — enforce configuration standards (e.g. "no public IPs on VMs")
- **Blueprints / template specs** — reproducible deployments with baked-in policies
- **RBAC** — control who can do what, at what scope
- **Resource locks** — protect critical resources from delete / modify
- **Tags** — organise for cost, ownership, environment

---

## Manageability

Two dimensions the exam distinguishes.

### Management **of** the cloud
How you interact with Azure resources.
- Azure Portal, Cloud Shell, CLI, PowerShell, mobile app
- ARM templates, Bicep for infrastructure as code
- Autoscale rules
- Alerts and automated responses

### Management **in** the cloud
How you configure Azure itself.
- Web-based portal for admin
- Command-line tools
- APIs
- Automation via templates and Bicep

## Gotcha checklist

- **Availability ≠ reliability.** Availability is uptime; reliability is fault-recovery.
- **Scalability ≠ elasticity.** Elasticity is *automatic* scalability.
- **Vertical scaling has a ceiling.** You cannot pick a VM SKU beyond what Azure offers.
- **Cloud does not automatically give you HA.** You must architect for it (multi-zone, multi-region, load balancers).
- **SLAs compound.** Two dependent 99.9% services in series ≈ 99.8% combined.

---

## Anchor facts to memorise

- Nine's ladder: 99% (3.65d), 99.9% (8.76h), 99.99% (52m), 99.999% (5m) downtime per year.
- Vertical = up/down (bigger box). Horizontal = out/in (more boxes).
- Elasticity = automation + scalability.
- Composite SLA is the *product* of dependent SLAs.
- Governance = *what* rules; RBAC = *who* has access.
