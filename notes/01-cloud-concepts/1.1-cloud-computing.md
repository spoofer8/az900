# 1.1 · Describe cloud computing

**Domain 1 · Cloud concepts** — 25–30% of exam

---

## What cloud computing is

Delivery of computing services — servers, storage, networking, databases, software, analytics, intelligence — over the internet ("the cloud"), on-demand, with pay-as-you-go pricing.

**Core properties**

| Property                | Meaning                                                                 |
| ----------------------- | ----------------------------------------------------------------------- |
| On-demand self-service  | Provision resources without human intervention from the provider        |
| Broad network access    | Available over standard networks to a wide range of clients             |
| Resource pooling        | Provider's resources shared across many customers (multi-tenant)        |
| Rapid elasticity        | Scale up or down quickly to match demand                                |
| Measured service        | Usage metered; you pay for what you consume                             |

---

## CapEx vs OpEx

| CapEx (Capital Expenditure)                | OpEx (Operational Expenditure)               |
| ------------------------------------------ | -------------------------------------------- |
| Up-front purchase of physical assets       | Ongoing operating cost, no up-front spend    |
| Depreciated over years                     | Deducted the year it's incurred              |
| Traditional on-prem datacentre model       | Cloud consumption model                      |
| Predictable but inflexible                 | Flexible but variable                        |

> **Exam-hot** — Cloud shifts spending from **CapEx → OpEx**. This alone drives most executive cloud-adoption decisions.

---

## Consumption-based (pay-as-you-go) model

- No up-front cost
- No infrastructure to purchase / manage
- Pay for what you use (compute-seconds, GB stored, GB transferred out)
- Stop paying when you stop using
- Egress (data leaving Azure) is billed; ingress (data coming in) is free

---

## Shared responsibility model

Splits security and operational responsibilities between the **customer** and the **cloud provider**. The line moves depending on the service type.

| Responsibility                        | On-prem  | IaaS     | PaaS     | SaaS     |
| ------------------------------------- | -------- | -------- | -------- | -------- |
| Information & data                    | Customer | Customer | Customer | Customer |
| Devices (mobile / PCs)                | Customer | Customer | Customer | Customer |
| Accounts & identities                 | Customer | Customer | Customer | Customer |
| Identity & directory infrastructure   | Customer | Shared   | Shared   | Provider |
| Applications                          | Customer | Customer | Shared   | Provider |
| Network controls                      | Customer | Customer | Shared   | Provider |
| Operating system                      | Customer | Customer | Provider | Provider |
| Physical hosts                        | Customer | Provider | Provider | Provider |
| Physical network                      | Customer | Provider | Provider | Provider |
| Physical datacentre                   | Customer | Provider | Provider | Provider |

> **Always the customer's job, no matter the service type:** information & data, devices, accounts & identities.
> **Always the provider's job, in cloud:** physical hosts, network, and datacentre.

### The responsibility line shifts leftward as you go IaaS → SaaS

```
                  ← You manage                              Azure manages →
   On-prem   ══════════════════════════════════════════════════════════════▏
             Data · Identity · Apps · Runtime · OS · Virt · Hosts · Network

   IaaS      ═══════════════════════════════════════▏
             Data · Identity · Apps · Runtime · OS   │  Virt · Hosts · Network

   PaaS      ══════════════════════▏
             Data · Identity · Apps  │  Runtime · OS · Virt · Hosts · Network

   SaaS      ══════════════▏
             Data · Identity  │  Apps · Runtime · OS · Virt · Hosts · Network
             ▲
             │ Data & identities — always yours, in every model
```

## Cloud deployment models

| Model         | Who owns / operates                                 | Where it runs                       | When to use                                                         |
| ------------- | --------------------------------------------------- | ----------------------------------- | ------------------------------------------------------------------- |
| **Public**    | Third-party provider (Azure, AWS, GCP)              | Provider's datacentres              | Fastest to deploy, lowest up-front cost, elastic capacity           |
| **Private**   | Single organisation (or provider on their behalf)   | On-prem or hosted, single tenant    | Regulatory / sovereignty needs, legacy hardware, high customisation |
| **Hybrid**    | Combination of public + private                     | Both, connected                     | Gradual migration, burst-to-cloud, keep sensitive data on-prem      |
| **Multi-cloud** | Two or more public providers                      | Multiple provider datacentres       | Avoid lock-in, best-of-breed services, resilience across providers  |

**Azure Arc** enables single-plane management across all four — it's how Microsoft answers hybrid and multi-cloud.

### At a glance

```
    Public                              Private
   ┌────────────────────────┐         ┌────────────────────────┐
   │  Provider's DC         │         │  Your DC (or hosted    │
   │  shared across many    │         │  just for you)         │
   │  tenants               │         │  single tenant         │
   └────────────────────────┘         └────────────────────────┘
    Lowest cost, most elastic         Highest control, most compliance-friendly


    Hybrid                             Multi-cloud
   ┌──────────┐  VPN / ER  ┌────────┐  ┌───────┐ ┌───────┐ ┌───────┐
   │ Private  │◀══════════▶│ Public │  │ Azure │ │  AWS  │ │  GCP  │
   │   DC     │            │   DC   │  └───────┘ └───────┘ └───────┘
   └──────────┘            └────────┘
   Public + private connected          Two or more public providers
                                        (Azure Arc coordinates all four)
```

### Trade-offs

| Model         | Cost       | Control    | Scalability | Compliance ease | Speed to deploy |
| ------------- | ---------- | ---------- | ----------- | --------------- | --------------- |
| Public        | Lowest     | Lowest     | Highest     | Depends         | Fastest         |
| Private       | Highest    | Highest    | Limited     | Easiest         | Slowest         |
| Hybrid        | Middle     | High       | High        | Flexible        | Middle          |
| Multi-cloud   | Variable   | Variable   | Highest     | Complex         | Middle          |

## Gotcha checklist

- **Ingress vs egress:** data *into* Azure is free; data *out* of Azure costs money and across regions costs more.
- **Public cloud ≠ insecure.** Isolation is logical (accounts, subscriptions, RBAC), not physical.
- **Private cloud ≠ on-prem.** A private cloud can run in a provider's DC, dedicated to you.
- **Multi-cloud ≠ hybrid.** Multi-cloud is two public providers; hybrid is public + private.
- **Shared responsibility line depends on the *service* type, not the *deployment* model.**

---

## Anchor facts to memorise

- Cloud = on-demand, self-service, pay-as-you-go IT.
- CapEx → OpEx is the classic executive framing.
- Ingress: free. Egress: paid.
- IaaS/PaaS/SaaS = a spectrum of responsibility, from customer-heavy to provider-heavy.
- Data & identities are *always* the customer's responsibility.
