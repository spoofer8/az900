# 1.3 · Describe cloud service types

**Domain 1 · Cloud concepts** — 25–30% of exam

Four categories: **IaaS**, **PaaS**, **SaaS**, and (as a special case of PaaS) **serverless**. The exam heavily tests which category a given service belongs to and which one solves a given scenario.

---

## The three (four) service types at a glance

| Type          | You manage                              | Provider manages                       | Analogy                          |
| ------------- | --------------------------------------- | -------------------------------------- | -------------------------------- |
| **IaaS**      | OS, runtime, apps, data                 | Hardware, virtualisation, network      | Renting a house — you furnish it |
| **PaaS**      | Apps, data                              | OS, runtime, hardware                  | Renting a serviced apartment     |
| **SaaS**      | Data (mostly just usage & configuration) | Everything                             | Staying at a hotel               |
| **Serverless** | Function code + configuration          | Everything, incl. capacity & scale     | Ordering delivery — you just eat |

---

## Detailed shared-responsibility split

| Layer                        | On-prem | IaaS | PaaS | SaaS |
| ---------------------------- | :-----: | :--: | :--: | :--: |
| Data                         | You     | You  | You  | You  |
| Application                  | You     | You  | You  | Provider |
| Runtime                      | You     | You  | Provider | Provider |
| Middleware / OS              | You     | You  | Provider | Provider |
| Virtualisation               | You     | Provider | Provider | Provider |
| Servers                      | You     | Provider | Provider | Provider |
| Storage                      | You     | Provider | Provider | Provider |
| Networking                   | You     | Provider | Provider | Provider |

> **Gotcha** — In IaaS, the *guest OS* is your responsibility (patches, antivirus, updates). Many exam questions test this specifically.

### The stack — same layers, different owners

                On-prem       IaaS         PaaS         SaaS

## IaaS — Infrastructure as a Service

The customer rents virtualised infrastructure — VMs, networks, storage — and manages everything from the guest OS up.

**Azure IaaS examples**
- Azure Virtual Machines
- Virtual Networks (VNet)
- Managed Disks
- Azure Load Balancer (arguably shared)

**When to choose**
- Lift-and-shift migration of existing servers
- Full control over the OS
- Legacy apps that can't be re-platformed
- Custom software with unusual OS dependencies
- Disaster recovery site

**Trade-offs**
- Most flexibility, but most operational burden
- You still patch, back up, harden, monitor the guest OS

---

## PaaS — Platform as a Service

The customer supplies application code and data; provider supplies the entire runtime platform.

**Azure PaaS examples**
- Azure App Service
- Azure SQL Database
- Azure Cosmos DB
- Azure Container Apps
- Azure Kubernetes Service (managed control plane — nodes are IaaS-adjacent)

**When to choose**
- Building or hosting web / mobile / API apps
- You want to focus on code, not infrastructure
- You need built-in scaling, load balancing, deployment slots
- Rapid dev environments for teams

**Trade-offs**
- Less control over runtime / OS choice
- Locked into provider's runtime versions and features
- Great for new (cloud-native) apps; harder retrofit for older ones

---

## SaaS — Software as a Service

Fully finished software, delivered over the internet on subscription. Customer manages only their data and users.

**SaaS examples (Microsoft's and others')**
- Microsoft 365 (Word, Excel, Outlook, Teams)
- Dynamics 365
- Salesforce
- Google Workspace
- Slack, Zoom, Dropbox

**When to choose**
- Standard productivity / business function needs (email, CRM, collab)
- You want fastest time to value, no development
- You accept the provider's feature set as-is

**Trade-offs**
- Least flexibility
- Rely entirely on vendor for uptime, security, features, updates

---

## Serverless

Not a fourth service type in strict cloud taxonomy — it's a *consumption model* usually implemented on top of PaaS.

**Defining traits**
- No server management by the customer
- Auto-scale to zero when idle; scale up instantly on demand
- Billed per execution / per resource used, not per hour of provisioned capacity
- Event-driven (HTTP, queue message, timer, blob upload)

**Azure serverless examples**
- Azure Functions
- Azure Logic Apps
- Azure Container Apps (scale-to-zero mode)
- Azure Event Grid
- Some Cosmos DB tiers (serverless throughput)

**When to choose**
- Bursty, unpredictable, or infrequent workloads
- Event-driven glue code
- APIs with variable load

### Serverless — event-driven flow

```
   Trigger events                       Execution                Side effects
   ─────────────────────             ────────────────           ───────────────────
   HTTP request        ─┐
   Queue message       ─┤                                        Write to DB
   Timer / cron        ─┼──▶  Azure Functions / Logic Apps  ──▶  Call another API
   Blob upload         ─┤       (scales 0 → N automatically)      Send email / message
   Cosmos DB change    ─┤       cold-start on first request
   Service Bus message ─┘

   Billed per execution · scale to zero at idle · no server to manage
```

## How to answer "which service type?" scenario questions

| Scenario cue                                                             | Answer     |
| ------------------------------------------------------------------------ | ---------- |
| "Need to move a 15-year-old Windows Server 2012 app without recoding"    | IaaS       |
| "Deploy a Node.js web app with auto-scale and CI/CD out of the box"      | PaaS       |
| "Company email and calendaring for 500 employees"                        | SaaS       |
| "Run a small script on new blob uploads, few times a day"                | Serverless |
| "Full control of the guest OS to install a specific driver"              | IaaS       |
| "Managed database — no OS to patch"                                      | PaaS       |
| "Need to be up and running in an hour, no dev time"                      | SaaS       |

## Gotcha checklist

- **Azure Virtual Machines = IaaS.** Not PaaS, even though they're managed.
- **Azure Kubernetes Service** — control plane is managed (PaaS-ish), agent nodes are VMs. Microsoft classifies AKS as PaaS.
- **Microsoft 365 is SaaS**, not PaaS. It's a *finished product*.
- **Serverless is a mode of PaaS**, not a separate customer-provider split.
- **Data is always customer-owned**, even in SaaS — you must still classify, protect, and back it up.

---

## Anchor facts to memorise

- IaaS: you patch the OS. PaaS: you don't. SaaS: you barely touch anything.
- Serverless: pay-per-execution, scale-to-zero, event-driven.
- Data & identities: always yours, regardless of type.
- App Service, SQL DB, Cosmos DB = PaaS. VMs = IaaS. M365 = SaaS. Functions = serverless (PaaS).
